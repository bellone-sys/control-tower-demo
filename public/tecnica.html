Fermopoint Control Tower — Architettura & Implementazione v0.12.0

Stack Tecnologico

Il frontend del Control Tower è una Single Page Application costruita interamente con tecnologie web standard. Non sono presenti backend proprietari: tutta la logica risiede nel browser e i dati persistono in localStorage.

Libreria / Strumento Versione Ruolo
React 18.3 UI framework — rendering dichiarativo, gestione stato con hook e Context API
Vite 6.4 Build tool e dev server — HMR, tree-shaking, bundling ottimizzato per produzione
Leaflet 1.9 Libreria mappe interattive — rendering tile OpenStreetMap, marker, polyline
React-Leaflet 4.x Wrapper React per Leaflet — componenti dichiarativi MapContainer, Marker, Polyline
CSS Custom Properties — Sistema di token semantici per temi light/dark; nessun CSS-in-JS né preprocessore
localStorage Web API Persistenza client-side per preferenze utente, lingua, tema, storico eventi e contratti
Nessuna dipendenza da framework di routing esterni: la navigazione tra le viste avviene tramite stato React (variabile activeView nel contesto applicativo principale).

Architettura Applicativa

Il Control Tower è una SPA React pura, senza server-side rendering. L'intera applicazione viene servita da un unico file index.html generato da Vite. La struttura delle cartelle segue una separazione per responsabilità:

CartellaContenuto
src/components/Componenti React riutilizzabili (Card, Modal, Map, TabPanel, ecc.)
src/contexts/Provider React Context: ThemeContext, I18nContext, TutorialContext
src/services/Service layer puro (nessun React): historyService, storageService
src/data/Stub data statici per demo: PUDO, filiali, giri, spedizioni, drivers
src/pages/Viste principali dell'app (Overview, Giri, Spedizioni, Scenari, PUDO, ecc.)
public/docs/Documentazione HTML statica (glossario, manuale, tecnica, ecc.)

Pattern architetturali

  • Context API — stato globale (tema, lingua, tutorial) condiviso via React Context senza Redux o altre librerie di stato.
  • Service layer — funzioni pure che incapsulano la logica di accesso a localStorage e la normalizzazione dei dati, separando le responsabilità dal layer UI.
  • Stub data — tutti i dati operativi (PUDO, giri, spedizioni) sono definiti in file .js statici in src/data/. Sostituibili con chiamate API reali senza modificare i componenti.
  • Componenti controllati — form e filtri utilizzano pattern controlled component per mantenere lo stato in React.
La demo non include autenticazione reale. Il modulo di login è puramente client-side e non deve essere utilizzato in ambienti di produzione senza integrare un Identity Provider.

Context API

L'applicazione utilizza tre context React globali, ciascuno con il proprio Provider e hook personalizzato. Tutti i Provider sono annidati nel componente radice App.jsx.

Context Stato gestito Hook esposto
ThemeContext Preferenza tema: light | dark | system; tema effettivo risolto (light / dark) useTheme() → { theme, preference, setPreference }
I18nContext Lingua corrente (it | en | fr | de); dizionari di traduzione; funzione di traduzione useI18n() → { lang, setLang, t }
TutorialContext Step tutorial corrente, stato completamento per sezione, flag tutorial attivo useTutorial() → { step, isActive, nextStep, resetTutorial }

I context sono indipendenti tra loro: non esiste dipendenza diretta fra ThemeContext e I18nContext. L'ordine di annidamento nel Provider tree non ha rilevanza funzionale.

Gestione del Tema

Il sistema di theming è basato interamente su CSS Custom Properties applicate all'elemento document.documentElement tramite l'attributo data-theme. Non viene generato CSS dinamico a runtime.

Flusso di inizializzazione (ThemeProvider)

  1. Lettura preferenza da localStorage (chiave fp_ct_theme); default: system.
  2. Se preferenza = system, risolve il tema effettivo tramite window.matchMedia('(prefers-color-scheme: dark)').
  3. Imposta document.documentElement.setAttribute('data-theme', resolvedTheme).
  4. Aggiunge listener MediaQueryList.addEventListener('change', ...) per aggiornare automaticamente il tema quando l'utente cambia le preferenze OS.

Internazionalizzazione

Il sistema i18n è implementato senza librerie esterne. I dizionari sono oggetti JavaScript annidati e le chiavi usano dot notation.

Struttura dei dizionari

// src/i18n/it.js (estratto)
export default {
  nav: {
    overview:    "Panoramica",
    giri:        "Giri",
    spedizioni:  "Spedizioni",
    scenari:     "Scenari",
    pudo:        "PUDO",
    filiali:     "Filiali",
  },
  header: {
    settings:    "Impostazioni",
    notifications: "Notifiche",
    logout:      "Esci",
  },
  common: {
    save:        "Salva",
    cancel:      "Annulla",
    confirm:     "Conferma",
    loading:     "Caricamento...",
    noData:      "Nessun dato disponibile",
  }
}

Hook useI18n()

// Utilizzo nei componenti
import { useI18n } from '../contexts/I18nContext'

function MyComponent() {
  const { t, lang, setLang } = useI18n()

  return (
    <button onClick={() => setLang('en')}>
      {t('header.settings')}  {/* → "Impostazioni" (se lang = 'it') */}
    </button>
  )
}

Comportamento della funzione t(key)

  • Naviga l'oggetto dizionario con i segmenti separati da ..
  • Se la traduzione è assente nel dizionario corrente, tenta il fallback sul dizionario italiano.
  • Se la chiave non esiste in nessun dizionario, restituisce la chiave stessa (es. "nav.overview"), evitando crash silenziosi.

Persistenza e rilevamento automatico

  • La lingua corrente è salvata in localStorage sotto la chiave fp_ct_language.
  • Al primo avvio (chiave assente), viene letta navigator.language: il prefisso a 2 caratteri viene confrontato con le lingue supportate (it, en, fr, de). Se non corrisponde, default: it.

Service Layer

Il service layer è composto da moduli JavaScript puri (nessun hook React) che incapsulano la logica di accesso ai dati persistenti. Il modulo principale è src/services/historyService.js.

historyService.js

  • STORAGE_KEY: fp_ct_history
  • MAX_EVENTS: 2000 — al superamento, gli eventi più vecchi vengono rimossi (FIFO).
FunzioneFirmaDescrizione
recordEvent() (eventData) → event Persiste un nuovo evento di modifica nello storico. Aggiunge id e timestamp automaticamente.
getEntityHistory() (entityType, entityId) → event[] Restituisce tutti gli eventi relativi a una singola entità (es. tutti i cambiamenti di un PUDO specifico).
getTypeHistory() (entityType) → event[] Restituisce tutti gli eventi per un tipo di entità (es. tutti i cambiamenti di tutti i PUDO).
getGlobalHistory() (limit?) → event[] Restituisce l'intero storico ordinato per timestamp decrescente. Supporta limite opzionale.
seedDemoHistory() () → void Popola lo storico con eventi fittizi per la demo, se lo storico è vuoto.

Persistenza localStorage

Tutte le chiavi usano il prefisso fp_ct_ (Fermopoint Control Tower) per evitare conflitti con altre applicazioni sulla stessa origine.

Chiave Tipo dato Struttura / Valori possibili
fp_ct_user JSON Object { id, name, email, role, avatar } — profilo utente autenticato
fp_ct_user_settings JSON Object { notifications: boolean, compactMode: boolean, defaultFiliale: string }
fp_ct_language String "it" | "en" | "fr" | "de"
fp_ct_theme String "light" | "dark" | "system"
fp_ct_menu_state JSON Object { collapsed: boolean, activeSection: string } — stato sidebar navigazione
fp_ct_contracts JSON Array Array di contratti PUDO modificati dall'utente: [{ pudoId, tier, scadenza, note }]
fp_ct_scenario_favorites JSON Array Array di ID scenario salvati come preferiti: ["F001", "F002"]
fp_ct_history JSON Array Array di eventi (max 2000) — struttura evento in sezione Service Layer
Tutti i valori sono serializzati con JSON.stringify() e deserializzati con JSON.parse(). In caso di JSON malformato (es. corruzione manuale), il service layer gestisce l'eccezione e ripristina il valore di default.

Importazione CSV

Il modulo di importazione spedizioni accetta file CSV con separatore virgola (,) e codifica UTF-8. La prima riga deve contenere gli header esatti indicati di seguito.

Campo CSV Tipo Obbligatorio Descrizione
id stringa Sì Identificativo univoco della spedizione (es. FP240001)
tipo enum Sì consegna | ritiro
destinatario stringa Sì Nome del destinatario (consegna) o mittente (ritiro)
pudoId stringa Sì Codice PUDO di destinazione/origine (es. IT10006)
pudoNome stringa No Nome commerciale del PUDO — usato come fallback se pudoId non risolve
peso_kg numero Sì Peso in kilogrammi con decimale (es. 1.5)
dim_l_cm numero No Larghezza in centimetri (intero)
dim_h_cm numero No Altezza in centimetri (intero)
dim_p_cm numero No Profondità in centimetri (intero)
priorita enum No normal | high — default: normal
data data Sì Data spedizione in formato ISO: YYYY-MM-DD (es. 2026-04-05)

Accessibilità WCAG 2.1 AA

Il Control Tower implementa le linee guida WCAG 2.1 a livello AA. Di seguito le implementazioni specifiche adottate.

ARIA roles e attributi

Contesto Implementazione ARIA
Navigazione a tab (viste) role="tablist" sul contenitore, role="tab" su ogni elemento, role="tabpanel" sul contenuto, aria-selected="true/false"
Dialog / Modal role="dialog", aria-modal="true", aria-label o aria-labelledby con titolo dialog, focus trap attivo
Notifiche live aria-live="polite" per toast informativi, aria-live="assertive" per errori bloccanti
Icone decorative aria-hidden="true" su tutti gli elementi SVG e icone ornamentali
Pulsanti icon-only aria-label descrittivo (es. aria-label="Chiudi dialog")
Navigazione sidebar aria-current="page" sul link corrispondente alla pagina attiva

Altre implementazioni

  • Skip link — link nascosto visibile a focus (:focus) che permette di saltare direttamente al contenuto principale, evitando la ripetizione della navigazione con screen reader.
  • focus-visible — gli stili di focus sono applicati tramite la pseudo-classe CSS :focus-visible per evitare outline non necessari con il mouse, garantendoli con tastiera e screen reader.
  • Contrasto colori — il rosso brand #DC0032 su sfondo bianco #ffffff raggiunge un rapporto di contrasto di 5.5:1, superiore al minimo WCAG AA di 4.5:1 per testo normale.
  • Navigazione da tastiera — tutti gli elementi interattivi sono raggiungibili via Tab; le dialog supportano Escape per la chiusura; i tab panel supportano le frecce direzionali.
La conformità WCAG 2.1 AA è verificata manualmente con NVDA (Windows) e VoiceOver (macOS) e automaticamente tramite axe-core in fase di sviluppo.

Build e Deploy

Comandi principali

ComandoRisultato
npm run dev Avvia il dev server Vite su http://localhost:5173/control-tower-demo/ con HMR attivo
npm run build Genera la build di produzione nella cartella dist/ (minificazione JS/CSS, tree-shaking, asset hashing)
npm run preview Serve localmente la cartella dist/ per test pre-deploy
npm run lint Esegue ESLint sul codice sorgente

Configurazione Vite

Il base path è impostato a /control-tower-demo/ in vite.config.js per garantire il corretto funzionamento su GitHub Pages: 

// vite.config.js
import { defineConfig } from 'vite'
import react from '@vitejs/plugin-react'

export default defineConfig({
  plugins: [react()],
  base: '/control-tower-demo/',
  build: {
    outDir: 'dist',
    assetsDir: 'assets',
  }
})

Deploy su GitHub Pages

  • Metodo 1 — Branch gh-pages: eseguire npm run build e caricare il contenuto di dist/ sul branch gh-pages. GitHub Pages servirà automaticamente da quel branch.
  • Metodo 2 — GitHub Actions: il workflow .github/workflows/deploy.yml esegue automaticamente build e deploy ad ogni push su main.
# .github/workflows/deploy.yml (estratto)
name: Deploy to GitHub Pages
on:
  push:
    branches: [main]
jobs:
  build-and-deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: '20' }
      - run: npm ci
      - run: npm run build
      - uses: peaceiris/actions-gh-pages@v3
        with:
          github_token: ${{ secrets.GITHUB_TOKEN }}
          publish_dir: ./dist
Poiché l'app usa React Router in modalità hash o gestione dello stato interno (senza server-side routing), non è necessario configurare rewrite rules sul server. L'unico file richiesto è index.html come entry point.
