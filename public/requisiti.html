Fermopoint Control Tower — Specifica completa dei requisiti di sistema v0.12.0

1. Obiettivo e Scope

1.1 Descrizione del Sistema

Fermopoint Control Tower è un'applicazione web di gestione operativa progettata per supportare il network di distribuzione DPD Italia / Fermopoint. Il sistema centralizza il monitoraggio e la pianificazione di spedizioni, giri di consegna, punti di ritiro (PUDO), flotta veicoli, filiali e contratti commerciali in un'unica interfaccia operativa.

Il sistema si posiziona come strumento di decision support per i responsabili di filiale e gli operatori logistici, consentendo di passare da una gestione reattiva a una proattiva grazie a KPI in tempo reale, scenari di ottimizzazione dei giri e un sistema di allerta per le eccezioni operative.

1.2 Contesto Aziendale

DPD Italia opera attraverso una rete capillare di filiali e punti di ritiro Fermopoint distribuiti su tutto il territorio nazionale. La gestione operativa coinvolge quotidianamente:

  • Centinaia di giri di consegna pianificati su base giornaliera
  • Migliaia di spedizioni tracciate dal ritiro alla consegna finale
  • Una flotta mista di veicoli propri e in subappalto
  • Una rete di PUDO (Pick-Up and Drop-Off) convenzionati su tutto il territorio
  • Contratti con corrieri partner e punti di ritiro soggetti a revisione periodica

L'integrazione con i sistemi legacy (AS/400) e con il motore di ottimizzazione OptimoRoute consente di importare dati reali e generare piani di consegna ottimizzati direttamente dalla Control Tower.

1.3 Obiettivi del Progetto

  1. Fornire una visione unificata delle operazioni giornaliere tramite dashboard KPI
  2. Ridurre i tempi di pianificazione dei giri attraverso scenari di ottimizzazione automatica
  3. Migliorare la visibilità sulle eccezioni operative (ritardi, mancate consegne, anomalie)
  4. Digitalizzare la gestione dei contratti con analisi AI dei documenti PDF
  5. Standardizzare i processi operativi tra le diverse filiali del network

1.4 Scope del Sistema

In scope: Dashboard operativa, gestione spedizioni, giri di consegna, ottimizzazione scenari, gestione PUDO (lista e mappa), flotta (veicoli, carburanti, manutenzioni), filiali, contratti PDF con AI, gestione utenti e permessi, eccezioni operative, report, impostazioni personalizzazione, cronologia modifiche.
Out of scope (versione corrente): Tracking real-time GPS dei veicoli, integrazione diretta con sistemi di pagamento, gestione HR del personale, modulo contabilità e fatturazione automatica.

2. Attori del Sistema

Il sistema prevede tre ruoli principali con diversi livelli di accesso e responsabilità operative.

Admin Accesso completo a tutte le funzionalità del sistema, inclusa la gestione degli utenti, configurazione globale e visualizzazione dei log di audit. Dashboard · Spedizioni · Giri · Scenari · PUDO · Flotta · Filiali · Contratti · Utenti · Eccezioni · Report · Impostazioni · Cronologia IT Manager, Responsabile Operations DPD Italia
Manager Accesso operativo completo alle funzionalità di pianificazione e monitoraggio. Può creare e modificare entità operative ma non gestisce utenti né configurazione di sistema. Dashboard · Spedizioni · Giri · Scenari · PUDO · Flotta · Filiali · Contratti · Eccezioni · Report Responsabile di filiale, Coordinatore logistico

3. Requisiti Funzionali

I requisiti funzionali sono identificati dal prefisso RF seguito da un numero progressivo. La priorità è classificata secondo la convenzione MoSCoW: MUST requisito obbligatorio, SHOULD altamente raccomandato, COULD desiderabile ma opzionale.

ID Descrizione Priorità Sezione App
RF001 Autenticazione utente: Il sistema deve consentire il login tramite email e password. La sessione deve essere persistita in localStorage e ripristinata automaticamente al riavvio dell'applicazione. Il logout deve cancellare tutti i dati di sessione. MUST Login / Sessione
RF002 Gestione sessione e timeout: La sessione deve scadere automaticamente dopo un periodo di inattività configurabile (default 8 ore). Al timeout, l'utente deve essere reindirizzato alla pagina di login con un messaggio esplicativo. SHOULD Login / Sessione
RF003 Panoramica KPI operativi: La dashboard principale deve esporre almeno i seguenti indicatori in tempo reale: numero spedizioni del giorno (totale, consegnate, in transito, anomalie), tasso di consegna al primo tentativo, numero giri pianificati vs eseguiti, PUDO attivi vs inattivi, veicoli disponibili vs in manutenzione. MUST Dashboard
RF004 Grafici e trend storici: La dashboard deve includere almeno un grafico temporale (ultimi 7/30 giorni) per i principali KPI, con possibilità di filtrare per filiale o periodo. SHOULD Dashboard
RF005 Import spedizioni via API: Il sistema deve esporre un endpoint REST per l'import batch di spedizioni da sistemi esterni (AS/400, TMS). Il payload deve supportare almeno i campi: codice tracking, mittente, destinatario, indirizzo, peso, volume, servizio, stato. MUST Spedizioni
RF006 Import spedizioni via file: L'utente deve poter importare spedizioni caricando file in formato CSV o Excel (.xlsx). Il sistema deve validare i dati, mostrare un'anteprima prima della conferma e segnalare le righe con errori. MUST Spedizioni
RF007 Gestione giri di consegna: Il sistema deve consentire la creazione, modifica ed eliminazione di giri di consegna. Ogni giro deve includere: data, filiale, veicolo assegnato, autista, lista spedizioni associate, orario di partenza stimato e stato (pianificato, in corso, completato). MUST Giri
RF008 Wizard creazione scenario: Il sistema deve guidare l'utente nella creazione di uno scenario di ottimizzazione tramite un wizard a 5 passi: (1) selezione filiale e data, (2) selezione spedizioni, (3) configurazione parametri OptimoRoute (finestre orarie, capacità veicoli, priorità), (4) anteprima mappa del piano generato, (5) conferma e salvataggio. MUST Scenari
RF009 Confronto scenari: Il sistema deve permettere il confronto side-by-side di due scenari alternativi, mostrando metriche differenziali (km totali, numero fermate, tempo stimato, costo carburante) per supportare la decisione del responsabile. SHOULD Scenari
RF010 Gestione PUDO — Lista: Il sistema deve presentare l'elenco completo dei punti PUDO con funzionalità di ricerca full-text, filtri per stato (attivo/inattivo/sospeso), tipo, cap e filiale di riferimento. Deve essere possibile esportare la lista in CSV. MUST PUDO / Lista
RF011 Gestione PUDO — Mappa interattiva: Il sistema deve visualizzare i PUDO su mappa (Leaflet) con marker colorati per stato e cluster automatico ad alta densità. Cliccando su un marker deve aprirsi un popup con le informazioni principali del PUDO e un link alla scheda dettaglio. MUST PUDO / Mappa
RF012 Check-in PUDO: Il sistema deve supportare il processo di check-in operativo del PUDO (visita periodica di verifica), registrando data, operatore, esito e note. La cronologia dei check-in deve essere accessibile dalla scheda del PUDO. SHOULD PUDO / Check-in
RF013 Gestione flotta — Veicoli: Il sistema deve gestire l'anagrafica dei veicoli con almeno: targa, tipo, portata (kg/m³), anno immatricolazione, filiale di assegnazione, stato (disponibile, in uso, in manutenzione, dismesso) e scadenze documenti (revisione, assicurazione, bollo) con notifica in prossimità della scadenza. MUST Flotta / Veicoli
RF014 Gestione flotta — Rifornimenti carburante: Il sistema deve registrare i rifornimenti di carburante per ogni veicolo (data, km percorsi, litri, costo, operatore) e calcolare automaticamente il consumo medio (l/100 km) per monitorare l'efficienza energetica della flotta. SHOULD Flotta / Carburanti
RF015 Gestione flotta — Manutenzioni: Il sistema deve gestire interventi di manutenzione programmata e straordinaria con: tipo intervento, data, km, officina, costo e allegati (fattura PDF). Deve essere possibile pianificare manutenzioni future con promemoria automatici. SHOULD Flotta / Manutenzioni
RF016 Gestione filiali: Il sistema deve gestire l'anagrafica delle filiali DPD/Fermopoint con: nome, codice, indirizzo, responsabile, numero veicoli assegnati, numero PUDO nel bacino e KPI di filiale (volume spedizioni, tasso consegna). Solo gli Admin possono creare o modificare filiali. MUST Filiali
RF017 Gestione contratti — Upload e visualizzazione PDF: Il sistema deve consentire l'upload di contratti in formato PDF, associandoli a un PUDO o filiale. I contratti devono essere visualizzabili inline (senza download obbligatorio) con metadati editabili (data stipula, scadenza, tipo contratto, stato). MUST Contratti
RF018 Analisi contratti con AI: Il sistema deve offrire la possibilità di sottoporre il testo di un contratto a un modello AI (Claude API) per estrarre automaticamente: clausole principali, condizioni economiche, scadenze, obblighi delle parti e rischi rilevanti. COULD Contratti / AI
RF019 Gestione utenti: Gli Admin devono poter creare, modificare, disattivare ed eliminare account utente. Ogni account deve avere: nome, email, ruolo (Admin/Manager/Operatore), filiale di appartenenza e stato attivo/inattivo. Deve essere possibile resettare la password dell'utente. MUST Utenti
RF020 Gestione eccezioni operative: Il sistema deve consentire la segnalazione, il tracciamento e la risoluzione di eccezioni operative (mancata consegna, danneggiamento, giacenza scaduta, PUDO non operativo). Ogni eccezione deve avere: tipo, priorità, spedizione/PUDO associato, note e stato (aperta, in gestione, risolta). MUST Eccezioni
RF021 Report operativi: Il sistema deve generare report configurabili per: spedizioni (per filiale, per periodo, per stato), performance PUDO (volumi ricevuti, tasso ritiro), utilizzo flotta (km percorsi, costi carburante, fermo tecnico). I report devono essere esportabili in formato PDF e CSV. SHOULD Report
RF022 Impostazioni — Lingua: Il sistema deve supportare la selezione della lingua dell'interfaccia tra italiano (IT), inglese (EN), francese (FR) e tedesco (DE). La preferenza deve essere salvata nel profilo utente e persistita tra le sessioni. SHOULD Impostazioni
RF023 Impostazioni — Tema: Il sistema deve offrire la scelta tra tema chiaro (light) e tema scuro (dark mode). La preferenza deve rispettare anche il setting di sistema (prefers-color-scheme) se non esplicitamente sovrascritta dall'utente. COULD Impostazioni / Dark Mode
RF024 Impostazioni — Tutorial interattivo: Il sistema deve includere un tutorial guidato (tooltip overlay) che accompagni i nuovi utenti nella scoperta delle funzionalità principali. Il tutorial deve essere riattivabile manualmente dalle impostazioni. COULD Impostazioni / Tutorial
RF025 Cronologia modifiche (Audit Log): Il sistema deve registrare tutte le operazioni di creazione, modifica ed eliminazione effettuate da qualsiasi utente, con: timestamp, utente, entità modificata, vecchio valore e nuovo valore. La cronologia deve essere consultabile dagli Admin con filtri per utente, entità e intervallo temporale. SHOULD Cronologia

4. Requisiti Non Funzionali

I requisiti non funzionali sono identificati dal prefisso RNF. Definiscono le qualità di sistema trasversali che devono essere rispettate indipendentemente dalla funzionalità specifica.

ID Categoria Descrizione e Criteri di Accettazione Priorità
RNF001 Performance Il caricamento iniziale dell'applicazione (First Contentful Paint) deve avvenire entro 2 secondi su connessione a banda larga standard (≥ 10 Mbps). Le transizioni tra pagine interne non devono superare 500 ms. Il rendering della mappa PUDO con fino a 500 marker deve completarsi entro 1 secondo. MUST
RNF002 Accessibilità L'interfaccia deve rispettare le linee guida WCAG 2.1 livello AA. In particolare: contrasto minimo testo/sfondo 4.5:1 (3:1 per testi grandi), tutti gli elementi interattivi devono essere raggiungibili via tastiera, i form devono avere label associate, le immagini devono avere attributo alt significativo. MUST
RNF003 Sicurezza sessione I token di autenticazione devono essere conservati in localStorage con crittografia applicativa. Le password non devono mai essere salvate in chiaro. Il sistema deve implementare protezione CSRF per le chiamate API. In produzione, tutte le comunicazioni devono avvenire esclusivamente su HTTPS. MUST
RNF004 Compatibilità browser Il sistema deve funzionare correttamente nelle ultime due versioni stabili dei seguenti browser: Google Chrome, Mozilla Firefox, Microsoft Edge, Apple Safari. Non è richiesto il supporto a Internet Explorer o browser con market share inferiore all'1% nel segmento enterprise europeo. MUST
RNF005 Responsività mobile L'interfaccia deve essere utilizzabile su dispositivi con schermo a partire da 768 px di larghezza (tablet). Per schermi inferiori a 768 px (smartphone), le funzionalità principali di consultazione (dashboard, lista spedizioni, mappa PUDO) devono rimanere accessibili in layout adattivo. Le funzionalità di inserimento dati complesse sono ottimizzate per desktop. SHOULD
RNF006 Internazionalizzazione Il sistema deve supportare l'internazionalizzazione (i18n) per le lingue: Italiano (IT, default), Inglese (EN), Francese (FR), Tedesco (DE). Tutte le stringhe dell'interfaccia devono essere esternalizzate in file di traduzione. Date e numeri devono essere formattati secondo la locale selezionata. La lingua può essere modificata senza ricaricare l'applicazione. SHOULD
RNF007 Persistenza dati locale Le preferenze utente (tema, lingua, filtri attivi, colonne visibili nelle tabelle) devono essere persistite in localStorage e ripristinate automaticamente alla sessione successiva. La dimensione totale dei dati persistiti non deve superare 5 MB per utente. MUST
RNF008 Estensibilità API L'architettura del sistema deve prevedere uno strato API REST documentato (OpenAPI 3.0) che consenta l'integrazione con sistemi terzi senza modifiche al frontend. Le API devono essere versionate (/api/v1/) per garantire la retrocompatibilità nelle evoluzioni future. SHOULD
RNF009 Dark Mode Il sistema deve fornire un tema scuro completo, applicato a tutti i componenti dell'interfaccia inclusi tabelle, form, sidebar, mappa e grafici. Il contrasto dei colori in dark mode deve rispettare i requisiti WCAG 2.1 AA. La transizione tra tema chiaro e scuro deve avvenire con animazione CSS di durata ≤ 200 ms. COULD
RNF010 Manutenibilità Il codice sorgente deve seguire una struttura modulare con separazione chiara tra componenti UI, logica di business e layer dati. La copertura dei test automatici (unit + integration) deve essere ≥ 70% per i moduli core (autenticazione, import spedizioni, gestione giri, scenari). Il progetto deve includere una pipeline CI/CD con lint, test e build automatici. SHOULD

5. Vincoli e Assunzioni

5.1 Ambiente Demo vs Produzione

Attenzione: La versione corrente del sistema è un ambiente demo. Tutti i dati visualizzati (spedizioni, giri, PUDO, veicoli, contratti, utenti) sono fittizi e generati a scopo dimostrativo. Non vi è alcuna connessione a sistemi reali DPD/Fermopoint in questa fase.

La transizione all'ambiente di produzione richiederà:

  • Configurazione delle credenziali reali per i sistemi integrati (AS/400, OptimoRoute, Claude API)
  • Migrazione e validazione dei dati anagrafici reali (PUDO, filiali, veicoli, utenti)
  • Test di carico e penetration test prima del go-live
  • Formazione degli utenti finali tramite il tutorial integrato e il manuale utente

5.2 Dati Fittizi

I dati presenti nell'ambiente demo sono generati con le seguenti convenzioni:

  • Codici tracking nel formato DPD-XXXXXXXX con sequenza numerica progressiva
  • Indirizzi relativi a comuni italiani reali ma associati a destinatari inesistenti
  • PUDO con denominazioni commerciali di fantasia (es. Tabaccheria Centrale, Edicola Roma)
  • Nomi utenti e autisti generati casualmente, non riferibili a persone reali
  • Contratti PDF con testo segnaposto (Lorem Ipsum) per test dell'analisi AI

5.3 Mock AS/400

L'integrazione con il sistema legacy AS/400 di DPD Italia è simulata tramite un layer mock che restituisce risposte predefinite per le seguenti operazioni:

  • Ricerca spedizione per codice tracking
  • Aggiornamento stato spedizione
  • Export lista spedizioni del giorno per filiale
  • Creazione giacenza e notifica al mittente
In produzione, il mock AS/400 sarà sostituito da chiamate REST verso il middleware di integrazione aziendale (ESB). La struttura del payload è già allineata con le specifiche del middleware DPD Italia (versione API 3.2).

5.4 Mock OptimoRoute

Il motore di ottimizzazione OptimoRoute è integrato nella funzionalità Scenari per la generazione di piani di giro ottimizzati. Nell'ambiente demo, le chiamate API a OptimoRoute sono intercettate da un mock che:

  • Simula un tempo di elaborazione realistico (3–8 secondi) per le richieste di ottimizzazione
  • Restituisce piani di giro plausibili basati sulle coordinate degli indirizzi forniti in input
  • Rispetta la struttura del payload reale OptimoRoute API v2 per facilitare la migrazione

La configurazione dei parametri OptimoRoute disponibili nel wizard (finestre orarie, capacità veicolo, numero massimo di fermate, velocità media, bilanciamento del carico) rispecchia le opzioni reali dell'API OptimoRoute e sarà pienamente funzionale in produzione una volta configurata la chiave API.

5.5 Vincoli Tecnologici

  • Frontend: L'applicazione è sviluppata con React (Vite) e non richiede un server applicativo dedicato; può essere deployata su qualsiasi CDN o hosting statico (es. Netlify, Vercel, AWS S3 + CloudFront).
  • Mappe: Il componente cartografico è basato su Leaflet con tile layer OpenStreetMap. In produzione è possibile sostituire il tile provider con una soluzione commerciale (Mapbox, HERE) per SLA garantiti e maggior dettaglio cartografico.
  • Analisi AI: L'analisi dei contratti utilizza le API di Anthropic Claude. Il feature richiede una chiave API valida e l'accesso alla rete internet dal browser dell'utente. Non vengono inviati dati sensibili senza esplicita conferma dell'utente.
  • Storage: In assenza di un backend dedicato nella versione demo, tutti i dati operativi sono gestiti in memoria (React state) e parzialmente persistiti in localStorage. In produzione sarà necessario un backend con database relazionale (PostgreSQL raccomandato) e un sistema di autenticazione centralizzato (OAuth 2.0 / SAML 2.0 per SSO aziendale DPD).

5.6 Assunzioni

  1. Gli utenti finali dispongono di un browser moderno aggiornato e di una connessione internet stabile (minimo 4 Mbps) nelle sedi operative.
  2. Il dato anagrafico dei PUDO (ragione sociale, indirizzo, coordinate GPS) è fornito da DPD Italia in formato strutturato (CSV o JSON) al momento del setup iniziale in produzione.
  3. La responsabilità della correttezza dei dati importati tramite file (spedizioni) è dell'utente che effettua l'upload; il sistema esegue validazione formale ma non semantica.
  4. Le notifiche di scadenza (documenti veicoli, contratti) sono gestite lato client tramite controllo delle date al login; in produzione sarà necessario un servizio di notifica backend (email/push).
