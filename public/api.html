API Reference

Fermopoint Control Tower — REST API per sviluppatori v1.0

Introduzione

Il sistema Fermopoint Control Tower espone una REST API che permette di integrare e automatizzare tutte le operazioni disponibili nell'interfaccia web. Tutte le risorse sono accessibili tramite chiamate HTTP standard con formato JSON.

Base URL

Produzione:  https://api.fermopoint.it/v1
Demo/Mock:   localStorage (simulazione lato client)
In ambiente demo tutte le chiamate API sono simulate lato client tramite mock in localStorage. I dati restituiti sono fittizi a scopo illustrativo. In produzione le stesse chiamate verranno instradate verso il backend reale.

Formato risposta

Tutte le risposte sono in formato JSON con header Content-Type: application/json. Le risposte di successo seguono la struttura:

{
  "data": { ... },       // payload principale
  "meta": {              // metadati opzionali (es. paginazione)
    "total": 142,
    "page": 1,
    "limit": 20,
    "pages": 8
  }
}

Versionamento

La versione dell'API è inclusa nel path base (/v1). Le versioni precedenti restano attive per almeno 12 mesi dopo il rilascio di una nuova major version. Le breaking change vengono sempre accompagnate da un incremento della versione major.

Paginazione

Gli endpoint che restituiscono liste supportano i query params page (default: 1) e limit (default: 20, max: 100). La risposta include sempre i campi total, page, limit e pages.

Rate Limiting

In produzione sono previsti limiti di 300 richieste/minuto per token autenticato. In caso di superamento viene restituito HTTP 429 con header Retry-After.

Autenticazione

L'API utilizza Bearer Token JWT. Il token va incluso in tutte le richieste autenticate nell'header HTTP: 

Authorization: Bearer <token>

Il token ha una durata di 8 ore. Alla scadenza è necessario eseguire nuovamente il login. Il campo expiresIn nella risposta del login indica la durata in secondi.

Endpoint

POST /auth/login Ottiene un token JWT

Autentica un utente e restituisce il token di accesso.

Request body:

{
  "email": "operatore@fermopoint.it",
  "password": "secret"
}

Response 200:

{
  "data": {
    "token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
    "user": {
      "id": "u_001",
      "email": "operatore@fermopoint.it",
      "ruolo": "operatore"
    },
    "expiresIn": 28800
  }
}
POST /auth/logout Invalida il token corrente

Revoca il token JWT corrente lato server. Richiede header Authorization.

Response 200:

{
  "data": { "message": "Logout effettuato con successo." }
}
GET /auth/me Dati dell'utente autenticato

Restituisce il profilo dell'utente associato al token corrente.

Response 200:

{
  "data": {
    "id": "u_001",
    "email": "operatore@fermopoint.it",
    "nome": "Mario Rossi",
    "ruolo": "operatore",
    "filialeId": "F001",
    "createdAt": "2025-01-15T08:00:00Z"
  }
}

Spedizioni

Gestisce il ciclo di vita delle spedizioni: importazione, ricerca e consultazione dei colli in transito presso i punti PUDO.

ParamTipoNote
pageintegeropzionale default 1
limitintegeropzionale default 20, max 100
tipostringopzionale consegna | ritiro
pudoIdstringopzionale codice PUDO es. IT10006
searchstringopzionale ricerca full-text su id/destinatario
sortstringopzionale campo di ordinamento es. data
orderstringopzionale asc | desc

Uno scenario raggruppa tutti i giri pianificati per una filiale e rappresenta la fotografia operativa in un determinato periodo. La creazione di uno scenario attiva l'ottimizzatore OptimoRoute.

GET /scenari Lista scenari

Cronologia

Il sistema registra tutte le modifiche alle entità principali (PUDO, Giri, Scenari, Filiali, Contratti). La cronologia è accessibile sia per singola entità (via endpoint dedicati) sia in modo globale.

GET /history Cronologia per entità specifica

Query params:

ParamTipoNote
entityTypestringrichiesto pudo | giro | scenario | filiale | contratto
entityIdstringrichiesto ID entità es. IT10006
limitintegeropzionale default 50

Esempio: GET /history?entityType=pudo&entityId=IT10006&limit=10

Response 200:

{
  "data": [
    {
      "id": "h_001",
      "entityType": "pudo",
      "entityId": "IT10006",
      "campo": "stato",
      "vecchioValore": "sospeso",
      "nuovoValore": "attivo",
      "utente": "Mario Rossi",
      "timestamp": "2026-03-20T10:15:00Z",
      "note": "Riattivato dopo lavori."
    }
  ]
}

Codici di Errore

L'API usa codici di stato HTTP standard. In caso di errore il body della risposta include sempre un oggetto error con codice, messaggio leggibile e dettagli aggiuntivi opzionali.

Schema errore

{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "Il campo 'email' non è valido.",
    "details": {
      "field": "email",
      "value": "not-an-email"
    }
  }
}

Codici HTTP

Codice Status Descrizione
200 OK Richiesta completata con successo.
201 Created Risorsa creata con successo. Il body contiene la nuova risorsa.
202 Accepted Richiesta accettata, elaborazione asincrona avviata. Vedi jobId.
400 Bad Request Il body della richiesta contiene errori di sintassi o campi non validi.
401 Unauthorized Token assente, scaduto o non valido. Eseguire nuovamente il login.
403 Forbidden L'utente autenticato non ha i permessi per questa operazione.
404 Not Found La risorsa richiesta non esiste o non è accessibile dall'utente.
422 Unprocessable Entity Sintassi corretta ma la logica di business non consente l'operazione.
429 Too Many Requests Rate limit superato. Attendere il tempo indicato in Retry-After.
500 Internal Server Error Errore imprevisto lato server. Contattare il supporto con il requestId.
In ambiente demo gli errori HTTP 4xx e 5xx sono simulati lato client e non riflettono necessariamente i comportamenti del backend di produzione.
